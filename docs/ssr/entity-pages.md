# Entity-Driven SSR Pages

Snapshot can render bunshot entity pages directly during SSR without requiring a separate
`snapshot.manifest.json` route definition for each page. Bunshot resolves the request,
loads entity data, and passes a `PageLoaderResult` into Snapshot's SSR renderer. Snapshot
then maps that abstract declaration onto its existing manifest component system and streams
HTML from the normal `PageRenderer` pipeline.

## Flow

1. Bunshot resolves a URL to a manifest-declared page and loads its data.
2. Bunshot calls `renderer.renderPage(result, shell, bsCtx)`.
3. Snapshot maps `result` into a `PageConfig`, route state, resources, and optional shell.
4. Snapshot renders that mapped page through `PageRenderer` and the registered component set.

The renderer does not introduce a new UI system. It translates bunshot declarations into
the same config objects used by `ManifestApp`.

## Page Mapping

| Bunshot page type  | Snapshot output                                                            |
| ------------------ | -------------------------------------------------------------------------- |
| `entity-list`      | `row` heading/actions, optional `filter-bar`, `data-table`                 |
| `entity-detail`    | heading/actions, one or more `detail-card` sections, related `data-table`s |
| `entity-form`      | heading/actions, `form`                                                    |
| `entity-dashboard` | heading, `row` of `stat-card`s, optional `chart`, optional `feed`          |
| `custom`           | delegated to `createReactRenderer().render()` using the handler ref        |

The field metadata from bunshot drives display, input, and column formatting. Snapshot
reuses existing component formats for enums, dates, booleans, JSON, and string arrays.

## Data Flow

Bunshot preloads the page data. Snapshot stores that payload in route-scoped manifest state
and points components at it with `from` refs such as `entityPageData.items` or
`entityPageData.stats.0`.

This gives SSR pages real server-rendered content while preserving the existing client-side
component contracts:

- `data-table` renders prefetched rows immediately and can still refetch later.
- `detail-card`, `stat-card`, `chart`, and `feed` read the same prefetched route state.
- resources generated by the mapper remain available for follow-up client interactions such
  as pagination, delete actions, or form submissions.

## Navigation And Shell

If bunshot includes `navigation` in the loader result, Snapshot maps it onto the existing
`nav` component and wraps the page in an SSR app shell.

- `shell: "sidebar"` renders a sidebar nav plus main content area.
- `shell: "top-nav"` renders a top navigation bar.
- `shell: "none"` renders the page without a shell wrapper.

## Renderer Support

`createReactRenderer()` supports both entity pages and bunshot custom handler pages.

- entity pages are mapped into a one-route synthetic manifest and rendered through
  `PageRenderer`
- custom pages are delegated back to the normal route-module render path

`createManifestRenderer()` supports entity pages by augmenting the existing manifest at
render time. It rejects bunshot `custom` page declarations because handler-ref rendering is
part of the React route renderer path.

## Caching Metadata

Both SSR renderers forward bunshot ISR metadata onto the Snapshot response:

- `x-bunshot-revalidate`
- `x-bunshot-tags`

They also copy bunshot page meta into the rendered head tags before streaming the response.

## Component Notes

Entity-driven rendering only uses already-registered Snapshot components. No special entity
components are required.

The main components involved are:

- `data-table`
- `detail-card`
- `form`
- `stat-card`
- `chart`
- `feed`
- `filter-bar`
- `row`
- `heading`
- `button`

## Current Constraints

- bunshot types are mirrored locally in `src/ui/entity-pages/bunshot-types.ts` until the
  published `@lastshotlabs/bunshot-ssr` types are imported directly.
- custom page delegation currently reconstructs an empty params/query match because the
  current bunshot resolved-page shape does not expose those values to Snapshot's
  `renderPage()` contract.
- manifest renderer mode does not support bunshot custom handler pages.

## Relationship To Snapshot Manifests

Entity-driven SSR pages are a manifest-first bridge, not a parallel feature. They let a
bunshot manifest declaration generate the same Snapshot manifest config you could have
written by hand. That keeps rendering, theming, action handling, and hydration on the
existing Snapshot runtime instead of introducing a second rendering stack.

For manifest-declared SSR routing features (nested layouts, slots, SSR middleware, and
convention fallbacks), see [manifest-routing.md](./manifest-routing.md).
